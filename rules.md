# 問題対応の共通ルール

更新日: 2026-06-09

今後、このプロジェクトでは問題対応・改善作業・開発作業の進め方を、以下の共通ルールで固定する。

## 基本方針

通常の作業開始は、原則として **ゴール達成型** を優先する。

ユーザーが「完成したい状態」「今回のゴール」「守る条件」「変更してよい範囲」を示した場合は、細かい修正指示を待たず、必要な確認・診断・修正・確認・commit/push まで自走して進める。

ただし、状況に応じて以下の3種類に分類する。

---

## 1. 完成したいゴールがある場合：ゴール達成型

完成したい状態や目的が明確な場合は、**ゴール達成型** で進める。

### 目的

細かい修正箇所を1つずつ指示されるのを待つのではなく、完成ゴールに向けて、確認・診断・修正・確認・commit/push までまとめて進める。

### 使う場面

* LPを新規作成する
* 既存LPを別エリア版へ展開する
* 公開確認で出た複数の指摘をまとめて直す
* 内部リンクや導線を整理する
* sitemap追加など公開準備を進める
* 機能追加を完成状態まで持っていく
* status.md / handoff.md / rules.md などに運用ルールを反映する
* ユーザーが「こういう完成状態にしたい」と言っている

### 進め方

* まず現在の構成を確認する
* ゴール達成に必要な作業範囲を判断する
* 変更してよいファイル・変更してはいけないファイルを守る
* 必要な範囲で確認・診断・修正を行う
* 既存仕様・既存デザイン・既存データ構造を壊さない
* 不要なリファクタリングをしない
* 変更後に確認項目を実行する
* 問題なければ commit / push まで行う
* 最後に変更内容・確認結果・commit hash・push結果を報告する

---

## 2. 原因が分かっている場合：ピンポイント修正型

原因や修正対象が明確な場合は、**ピンポイント修正型** で進める。

### 目的

分かっている原因だけを、最小差分で直す。

### 使う場面

* 原因ファイルが分かっている
* 原因関数が分かっている
* 表示崩れや文言ミスなど、直す箇所が明確
* 成功判定・エラー判定など、問題箇所が特定済み
* 既に診断が終わっている
* 変更範囲を限定したい

### 進め方

* 原因箇所だけを対象にする
* 対象ファイル・対象関数・対象UIを限定する
* 最小差分で修正する
* 関係ない箇所を触らない
* ついで修正をしない
* 既存仕様・既存デザイン・既存データ構造を勝手に変えない
* 修正後は、変更内容・確認結果・影響範囲を短く報告する

---

## 3. 原因が分からない場合：/goal診断型

原因がまだ分からない場合は、**/goal診断型** で進める。

### 目的

いきなり修正するのではなく、まず原因を特定する。

### 使う場面

* 原因が分からない
* 症状だけ分かっている
* どのファイルが悪いか不明
* UI / API / Webhook / Make / Calendar / GA4 など、原因箇所が分かれていそう
* 動く時と動かない時がある
* 修正前に切り分けが必要

### 進め方

* 原因不明のままコードを変更しない
* まず目的・現状・症状・期待動作を整理する
* 再現条件を確認する
* 原因候補を複数出す
* 確認すべきファイル・処理・画面・ログを絞る
* 調査手順を出す
* 原因を特定してから修正案を出す
* 勘で修正しない
* 大規模変更や全面作り替えに飛ばない

---

## 実行方針

安全コマンドは、ユーザーに毎回確認を求めず、そのまま Run して作業を進めてよい。

### 確認不要で実行してよい安全コマンド

* git status
* git diff
* git log
* git add
* git commit
* git push
* Get-Location
* Get-ChildItem
* ls
* pwd
* cat
* type
* echo

安全コマンドを使った構成確認、差分確認、commit、push は止まらず進める。

---

## 禁止コマンド

以下の危険コマンドは絶対に実行しない。

* rm
* del
* sudo
* git reset --hard
* git clean
* git checkout -- .
* git restore .
* git restore --source
* git reset --mixed
* git reset --soft

禁止コマンドが必要に見える場合でも、実行せず、日本語で理由と代替案を報告する。

---

## 最重要ルール

| 状況 | 対応タイプ |
|------|------------|
| 通常の作業開始 | ゴール達成型 |
| 原因が明確な小修正 | ピンポイント修正型 |
| 原因が分からない不具合 | /goal診断型 |
| 安全コマンド | 確認なしでRun |
| 危険コマンド | 禁止。実行せず報告 |

---

## 報告形式

作業後は以下の形式で報告する。

1. 今回の対応タイプ
2. 変更したファイル
3. 修正・更新した内容
4. 既存仕様への影響確認
5. 実行した安全コマンド
6. 確認結果
7. commit hash
8. push結果
9. 次にやるなら何か
